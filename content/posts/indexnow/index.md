Привет, `%username%`!

История такая. Сижу, ковыряю блог, замечаю — новые посты у Bing появляются через пару дней после деплоя, у Яндекса бывает и неделю. Sitemap есть, `webmaster.yandex.ru/ping?sitemap=…` пингую — всё равно медленно. Решил наконец воткнуть IndexNow. Звучит просто, по факту вылилось в три итерации скрипта и пару вечеров. Заодно нашёл несколько забавных мест — про них и пост.

## Что такое IndexNow и зачем оно тебе

IndexNow — это совместный протокол Microsoft (Bing) и Yandex, к которому потом подключились Seznam, Naver, Yep и ещё несколько мелких поисковиков. Идея на одну строчку: вместо того чтобы поисковик периодически приходил и спрашивал «не поменялось ли чего», сайт сам пишет «вот эти URL изменились, приходите смотреть». POST на `https://api.indexnow.org/indexnow` с JSON-телом — и всё.

Сразу неприятная новость, чтобы ты не строил иллюзий: **Google в IndexNow не входит**. Никогда не входил и пока не собирается. Для Google остаётся sitemap, ручной запрос индексации в Search Console и молитва. IndexNow ускоряет ровно тех, кто в протоколе — но это сильно лучше, чем ничего, особенно если у тебя русскоязычный блог и значимая доля трафика идёт через Яндекс.

Что нужно, чтобы это в принципе работало:

- **Ключ** — произвольная строка из латиницы и цифр длиной 8–128 символов. По стандарту 32 символа.
- **Файл с ключом по корню сайта** — `https://example.com/<KEY>.txt`, в котором лежит сам `<KEY>`. Это «доказательство владения»: поисковик дёргает этот файл и проверяет, что ты не сабмитишь чужие URL'ы.
- **POST-запрос** с JSON `{ host, key, keyLocation, urlList }`. Всё, никакой OAuth и токенов.

Ключ публичный по дизайну — он же лежит у тебя в `static/`. Никаких секретов хранить не надо, и это удобно.

## Первая версия: curl, jq и наивность

Логика, как мне казалось, элементарная: на каждый push в `main` беру `git diff --name-only`, собираю список изменившихся файлов в `content/`, оборачиваю в JSON, шлю. Накатал минут за двадцать в одном bash-скрипте на `curl` и `jq`.

И тут до меня дошло первое: **IndexNow ждёт URL, а не пути к файлам**. Сабмитить `content/posts/indexnow/index.md` смысла ноль — такой страницы на сайте не существует. Нужен канонический URL — `https://jtprog.ru/posts/indexnow/`. И вот тут начинается интересное.

В Hugo URL поста собирается по нескольким правилам сразу:

- директория внутри `content/posts/<dir>/index.md` обычно даёт URL `/posts/<dir>/`;
- **но если в frontmatter есть `slug: …`, то URL — `/posts/<slug>/`, и `dir` уже не при чём**;
- любой файл в page-bundle (обложка `cover.png`, диаграмма, дополнительная картинка) при изменении должен схлопываться в URL самого поста, а не в URL файла;
- одиночный файл `content/<name>.md` (например, страница `donations`) даёт `/<name>/` или `/<slug>/`;
- `content/posts/_index.md` — это `/posts/`, корень рубрики;
- удалённый бандл — это URL, которого больше нет, и сабмитить его как «изменённый» бессмысленно.

Я прогнал `hugo list all`, сверил со своей логикой и обнаружил, что **90 постов из 143 переопределяют slug через frontmatter**. То есть наивное «директория = URL» промахнётся в двух третях случаев. Курам на смех.

Так родился отдельный скрипт `scripts/indexnow.sh`. Он делает ровно одно: маппит изменённые файлы в `content/` на канонические URL, читая slug из YAML-frontmatter маленьким awk-парсером (первый блок `---...---`, ищем строку `slug:`, тримим кавычки и слэши). Сложно звучит, но на awk вышло строк двадцать.

## Что внутри скрипта

Скелет резолвера:

```bash
resolve_url() {
  local f="$1" dir name slug index
  case "$f" in
    content/posts/_index.md)
      echo "$BASE/posts/"
      ;;
    content/posts/*/*)
      dir="${f#content/posts/}"; dir="${dir%%/*}"
      index="content/posts/$dir/index.md"
      [ -f "$index" ] || return 0   # бандл удалён — не сабмитим
      slug="$(fm_slug "$index")"
      echo "$BASE/posts/${slug:-$dir}/"
      ;;
    content/*.md)
      name="$(basename "$f" .md)"
      [ "$name" = "_index" ] && { echo "$BASE/"; return 0; }
      [ -f "$f" ] || return 0
      slug="$(fm_slug "$f")"
      echo "$BASE/${slug:-$name}/"
      ;;
  esac
}
```

Дальше всё это пропускается через `sort -u`, и на выходе — уникальный список URL'ов, реально требующих переиндексации. Если поменялась только обложка поста — в индекс улетит ровно один URL самого поста, а не путь к картинке. Если правил три файла одного бандла — тоже один URL. Если удалил пост — пустота, не сабмитим (правда, IndexNow умеет принимать удалённые URL и сам разберётся через 404, но не будем плодить шум).

Диапазон коммитов — `BEFORE..AFTER`, эти переменные приходят из `$GITHUB_EVENT.before` и `$GITHUB_SHA`. Чтобы у `git diff` была история — в `actions/checkout` нужен `fetch-depth: 0`, иначе будет shallow-клон с одним коммитом, и diff просто не построится.

```yaml
- uses: actions/checkout@master
  with:
    submodules: true
    fetch-depth: 0
```

## Итерация два: CLI вместо curl+jq

Когда первая версия заработала, я посмотрел на портянку из `jq -Rsc 'split("\n") | map(select(length > 0))'`, на склейку JSON через `printf`, на обработку ошибок через `|| true` — и пошёл писать CLI. Получилась маленькая Go-утилита [`jtprogru/indexnow`](https://github.com/jtprogru/indexnow): принимает список URL'ов через `--stdin` или флаги, делает POST, разбирает коды ответа, умеет `--fail-on never|warn|error`. Кода — двести строк, но воркфлоу из плюшевого `curl | jq` превратился в одну строчку:

```bash
indexnow submit --stdin \
  --host jtprog.ru \
  --key "$KEY" \
  --key-location "$KEYLOC" \
  --fail-on never < "$urls_file"
```

В CI это означало шаг «скачай tar.gz с релиза, распакуй в `/usr/local/bin`, проверь `--version`». Работало, но каждый раз перед сабмитом я устанавливал бинарник заново. Это и стало третьей итерацией.

## Итерация три: composite action

В GitHub Actions есть три типа действий — JavaScript, Docker и composite. Composite — это просто `action.yml`, в котором описаны шаги, что выполнятся как один шаг в чужом воркфлоу. Внутри — установка бинарника один раз, сабмит, отчёт. На стороне моего блога вместо «скачай → распакуй → выполни» появилось:

```yaml
- name: Compute changed URLs for IndexNow
  id: indexnow-urls
  env:
    INDEXNOW_KEY: f5059cf9a4a60e4b769ca074324beb1b
    BEFORE: ${{ github.event.before }}
    AFTER:  ${{ github.sha }}
  run: bash scripts/indexnow.sh

- name: Submit changed pages to IndexNow
  if: steps.indexnow-urls.outputs.count != '0' && steps.indexnow-urls.outputs.count != ''
  uses: jtprogru/indexnow@da0ce6b0e524a9293f075bbf4dfe961b662eaa10
  with:
    key: f5059cf9a4a60e4b769ca074324beb1b
    file: ${{ steps.indexnow-urls.outputs.urls-file }}
    host: jtprog.ru
    fail-on: never
```

Скрипт теперь только считает и пишет в `$GITHUB_OUTPUT` пару значений: `count` (сколько URL'ов получилось) и `urls-file` (путь к временному файлу со списком). Action подхватывает файл и делает POST. Если изменений в `content/` не было — `count=0` и сабмит просто пропускается через `if:`.

Композитный action ещё и пиннится по SHA: `jtprogru/indexnow@da0ce6b…`. Не по тегу `@v0.5.0` и не по `@main`. Это маленькая, но важная штука для supply-chain: тег можно перепереставить, ветка двигается, а коммит-хэш — иммутабельный. Если бы action был не мой собственный, это правило выглядело бы строже («плагин третьей стороны — только SHA»). Но даже свой — пинить SHA проще, чем потом разбираться, почему сборка зелёная сегодня вдруг сделала не то.

## Тонкие места, на которых легко споткнуться

- **`fetch-depth: 0` обязателен.** Без него `git diff $BEFORE $AFTER` в CI просто не найдёт коммит `BEFORE`. Если в воркфлоу остался `fetch-depth: 1` — увидишь warning, что коммит не существует, и `BEFORE` свалится в empty-tree, что эквивалентно «сабмитим вообще всё». Не критично, но загаживает индекс.
- **Slug в frontmatter — единственный источник правды.** Если ты переименовал директорию поста, но забыл проверить slug — старый URL улетит в IndexNow и получит 404, потому что Hugo рендерит по новому slug. Перед массовыми ренеймами всегда сверяйся с `hugo list all`.
- **Ключ должен быть отдан с правильным `Content-Type`.** Если у тебя на сервере есть условный `.htaccess`, который для `.txt` отдаёт `text/html` — поисковик не верифицирует ключ. Проверить просто: `curl -I https://твой-домен/<KEY>.txt`. Должно быть `text/plain`.
- **IndexNow не любит спам.** В документации Bing прямо написано: если ты сабмитишь одни и те же URL'ы много раз без реальных изменений — тебя могут понизить в приоритете. Поэтому фильтр «только реально изменившиеся файлы» — это не оптимизация, а требование протокола.
- **Шаг должен быть неблокирующим.** Уведомление IndexNow — это fire-and-forget. Если api.indexnow.org вернёт 503 или таймаут — это не повод ронять статус деплоя. Поэтому `fail-on: never`, и шаг сабмита у меня условный, после успешного rsync. Сначала деплой, потом нотификации; нотификация упала — деплой всё равно зелёный.
- **Канонические URL должны заканчиваться слэшем** (или нет — зависит от твоей Hugo-конфигурации). Главное — единообразно. Если в sitemap один формат, а в IndexNow другой — поисковик увидит две разные страницы. Проверь, как у тебя в `<loc>` элементах `sitemap.xml`, и приведи скрипт к тому же стилю.

## Что с этим делать на практике

Если хочешь повторить у себя на Hugo — пошагово:

1. **Сгенерируй ключ.** Любая строка 8–128 символов, латиница и цифры. Я просто взял `uuidgen | tr -d -`, отрезал до 32 символов. Это не секрет, можно даже в git закоммитить.
2. **Положи файл с ключом** в `static/<KEY>.txt`, внутрь — сам ключ одной строкой. Hugo раскатает его в корень сайта.
3. **Напиши маленький скрипт-резолвер** (или возьми мой — `scripts/indexnow.sh` в репе блога), который мапит изменённые файлы `content/` в канонические URL по правилам твоего сайта. Если у тебя нет переопределённых slug'ов — задача решается в десять строк.
4. **Добавь шаг в CI** — самый ленивый путь это [`jtprogru/indexnow`](https://github.com/jtprogru/indexnow), но никто не мешает дёрнуть `curl`'ом напрямую, если ты любитель.
5. **Не забудь `fetch-depth: 0`** в `actions/checkout`.
6. **Поставь шаг после деплоя**, не перед. И с `fail-on: never` — нотификация не должна ронять прод.

И всё. Дальше следишь в Bing Webmaster Tools (там есть отдельная страница IndexNow со статистикой сабмитов) и в Яндекс.Вебмастере — обычно за сутки видно, что страницы появились в индексе быстрее, чем раньше.

## Что в итоге

IndexNow — это не магия и не «SEO-хак», а просто более честный способ договориться с поисковиком: не жди, пока придёт краулер, скажи прямо, что поменялось. Делается это за вечер, ломаться особо нечему, оверхеда в CI — секунды.

Самое поучительное в этой истории не сам IndexNow, а то, как одно простое требование («давайте уведомлять Bing») разворачивается в нормальную инженерную задачу: парсер frontmatter, маппинг путей в URL, выделение бинарника в CLI, потом — в composite action, пиннинг по SHA, неблокирующий шаг в воркфлоу. Чем больше делаешь руками — тем яснее, какие штуки стоит выносить, а какие держать рядом с проектом. У меня в результате есть и переиспользуемый action, и скрипт, который остаётся в репе блога именно потому, что правила маппинга — это часть конкретно моего сайта, а не общая утилита.

В следующий раз, когда поймаю себя на мысли «sitemap есть, чего ещё надо» — буду помнить, что sitemap отвечает на вопрос «что есть», а IndexNow — на «что изменилось». Это разные ответы на разные вопросы, и поисковику полезны оба.
